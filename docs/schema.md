# QnEvt Proof Schema

QnEvt proofs are portable JSON records that describe how a random output was produced and how it can be checked later. The schema is shared by Secret Generator, Audit Sampling, and Fair Random workflows.

The current production schema version is `1.0`.

## Proof Object

```json
{
  "version": "1.0",
  "proof_type": "secret_generation",
  "input_hash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  "entropy_batch": "qnb_0123456789abcdef",
  "mixing_algorithm": "XOR-SHA256 (Local CSPRNG + QnEvt)",
  "output_hash": "b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9",
  "timestamp": "2026-05-29T12:00:00Z",
  "signature": "unsigned_client_side"
}
```

## Fields

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `version` | string | yes | Proof schema version. Current value: `1.0`. |
| `proof_type` | string | yes | One of `secret_generation`, `audit_sampling`, or `fair_random`. |
| `input_hash` | string | yes | SHA-256 digest of the locked input or generation metadata. |
| `entropy_batch` | string | yes | Entropy batch identifier. Live batches use `qnb_` followed by 16 lowercase hex characters. Local-only proofs may use `local_simulation`, `qnevt_live_pool`, or `unknown`. |
| `mixing_algorithm` | string | yes | Human-readable algorithm identifier used to derive the output. |
| `output_hash` | string | yes | SHA-256 digest of the final secret, sampled CSV, or winner list. |
| `timestamp` | string | yes | ISO 8601 timestamp generated by the client. |
| `signature` | string | optional | Current browser-side proofs use `unsigned_client_side`; future node-signed proofs should contain a detached signature. |

## Type-Specific Inputs

### `secret_generation`

`input_hash` commits to generation metadata, not the secret itself. The raw secret must never be uploaded to QnEvt.

Recommended metadata fields:

- `proof_type`
- `secret_type`
- `output_encoding`
- `byte_length`
- `entropy_batch`
- `mixing_algorithm`

`output_hash` is the SHA-256 digest of the generated secret string.

### `audit_sampling`

`input_hash` commits to the normalized source dataset and sampling parameters. The current browser implementation normalizes CSV line endings before hashing.

`output_hash` is the SHA-256 digest of the sampled CSV output.

### `fair_random`

`input_hash` commits to the participant list order and winner count.

`output_hash` is the SHA-256 digest of the selected winners in order, joined by commas.

## Verification Expectations

A verifier should:

1. Parse the proof JSON and validate all required fields.
2. Recompute `input_hash` from the original input and parameters.
3. Resolve `entropy_batch` through the QnEvt batch registry when it is a live `qnb_...` batch.
4. Re-run the declared deterministic algorithm using the committed input and entropy material.
5. Recompute `output_hash` and compare it with the proof.
6. Check `signature` when node-signed proofs are available.
